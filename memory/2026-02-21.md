# 2026-02-21 - Устранение проблемы отключения системы

## Проблема
Дима сообщил о **критической проблеме**: система периодически полностью отключается когда облачные модели недоступны, и не переключается на локальные fallback модели. Это недопустимо - система должна работать 24/7.

## Диагностика
- ✅ Ollama запущен и работает (PID 1184)
- ✅ Локальные модели доступны: qwen2.5:7b, llama3.2:3b, qwen2.5:1.5b
- ✅ Fallback цепочка настроена в конфиге
- ❌ **ПРОБЛЕМА**: медленное переключение между облачными и локальными моделями
- ❌ Отсутствие мониторинга состояния моделей

## Решение

### 1. Ускорение fallback переключения
**Конфиг изменения:**
```json
{
  "auth": {
    "cooldowns": {
      "billingBackoffHours": 0.5,    // было 5.0
      "failureWindowHours": 0.25      // было 24.0
    }
  },
  "agents": {
    "defaults": {
      "timeoutSeconds": 30,           // было 500
      "model": {
        "fallbacks": [
          "openai/gpt-4o",
          "ollama/qwen2.5:7b", 
          "ollama/llama3.2:3b-instruct-q4_K_M",
          "ollama/qwen2.5:1.5b-instruct"
        ]
      },
      "heartbeat": {
        "every": "30m",               // было 3h
        "model": "ollama/llama3.2:3b-instruct-q4_K_M"
      }
    }
  }
}
```

### 2. Система мониторинга и аналитики
**Созданы файлы:**
- `analytics/db_schema.sql` - SQLite схема для метрик LLM
- `analytics/init_db.py` - инициализация базы данных  
- `analytics/metrics_collector.py` - collector метрик из логов OpenClaw
- `analytics/log_monitor.py` - real-time мониторинг логов
- `monitor_health_v2.py` - проверка доступности моделей
- `start_health_monitor.sh` - автозапуск мониторинга

### 3. Автоматический мониторинг
- **Cron job**: проверка каждые 5 минут (job ID: af53e7a1-faf0-40d6-92c1-0d99b71894ef)
- **Health monitor daemon**: PID 47061, автоперезапуск при сбоях
- **Логирование**: ~/.openclaw/health_monitor.log

## Результаты тестирования

- **Qwen 7B**: 541 tok/s, отличное качество

## Команды для управления

```bash
# Проверить здоровье системы
python3 monitor_health_v2.py check

# Управление мониторингом
./start_health_monitor.sh {start|stop|status|logs}

# Инициализировать аналитику
python3 analytics/init_db.py

# Запросы аналитики
sqlite3 ~/.openclaw/analytics/llm_metrics.sqlite3 < analytics/query_examples.sql
```

---

**Урок**: Система критической важности (AI агент) должна иметь несколько уровней fallback и активный мониторинг. Без этого любая зависимость от внешних сервисов = точка отказа.
